# Klako: Product Vision & Strategic Roadmap

*Date: May 2026*

## The Vision: The Autonomous Principal Engineering Workforce
Klako is no longer a conversational "coding assistant." It is an **Autonomous Principal Engineering Workforce**. 

Where competitors (like Claude Code) focus on single-turn terminal interactions, Klako is designed for **sustained, unsupervised orchestration**. We do not chat; we deploy specialized subagents to solve complex, multi-layered engineering and analytical problems in parallel.

### The Three Pillars of Klako

#### 1. The Swarm Engine (`/loop`)
The core mechanic is the `SwarmOrchestrator`. A user provides a high-level objective (`/loop De-risk the authentication module`). Klako autonomously:
1.  **Decomposes** the goal into atomic tasks via a Tier-1 Architect.
2.  **Spawns** specialized, parallel subagents (Engineers, QA, Data Analysts) via the `tools::Delegate` infrastructure.
3.  **Validates** outcomes rigorously before declaring the loop complete.

#### 2. Deterministic "Worlds" (The Anti-Hallucination Layer)
LLMs are creative but mathematically and logically flawed. Klako leapfrogs pure-LLM agents by routing domain-specific logic to **high-performance, deterministic Rust kernels** ("Worlds").
*   Instead of guessing a physics conversion, agents query `PhysicsWorld`.
*   Instead of hallucinating a Monte Carlo distribution, agents trigger `ActuarialWorld` or `MarketWorld` to run 10,000 trials in Rust.
*   Instead of guessing structural parity, `ParityWorld` enforces rigid style constraints via AST parsing.

#### 3. The High-Fidelity Command Center (UI Strategy)
A terminal is a bottleneck for swarm orchestration. Klako's primary interface is shifting to a local, browser-based **Command Center** designed for observability and steering:
*   **Swarm Sprint Board:** A live Kanban board (Open, In Progress, Done) visualizing the parallel execution of the `SwarmOrchestrator`.
*   **Telemetry Canvas:** A GPU-accelerated canvas streaming mechanical logs and execution telemetry without freezing the DOM.
*   **Steerable Plan Editor:** A side-pane that loads `.kla/sessions/PLAN.md`, allowing the human operator to edit the swarm's trajectory mid-flight.
*   **Visual Artifacts:** Native rendering of Data Visualizations (Gantt Charts, Histograms) generated by the "Worlds."

## Roadmap: Next Strategic Milestones

### Milestone 1: UI Orchestration & The Command Center
*   **Sync Orchestrator to UI:** Update the `SwarmOrchestrator` to emit `SwarmLedgerUpdate` WebSocket events, breathing life into the existing HTML/JS Sprint Board.
*   **Plan Mode Activation:** Wire the Plan Editor to intercept the Architect's decomposition, allowing human approval/modification before execution begins.
*   **Visual Artifact Rendering:** Ensure that when a World tool yields a `VisualArtifact` (e.g., a Gantt chart from `LogisticsWorld`), it renders natively in the web UI.

### Milestone 2: Verification and "Outcomes"
*   **Formal Outcomes:** Update the `SwarmOrchestrator` to require explicit, empirical proof of success. A task isn't "Complete" unless a deterministic tool (like `ParityWorld`, test runners, or `DiscoveryWorld`) verifies the state change.

### Milestone 3: The `/dream` Mechanic (Continuous Improvement)
*   **Asynchronous Retrospectives:** Implement a background task (or `/dream` command) that analyzes `logs.json` and recent `.kla/sessions/` to identify recurring failures.
*   **Axiom Generation:** The Dream process will auto-propose improvements to `KLA.md` or dynamically generate new `SKILL.md` profiles to prevent future errors.
